## Pre-Deployment Checklist

Before starting deployment, verify:

- [ ] All quality gates pass (`test`, `test:contract`, `lint`, `typecheck`).
- [ ] `CHANGELOG.md` is updated with the new version.
- [ ] Version numbers are consistent across `package.json` and `wxt.config.ts` manifest.
- [ ] No `.env.local` or secrets are included in the build output.
- [ ] A smoke test has been performed on the built extension.

## Build Process

### Production Build for Chrome

```powershell
# Navigate to the project root
# Install dependencies (if not already done)
pnpm install

# Build the extension
pnpm --filter @the-abyss/sentra-assist build

# Package as ZIP for store submission
pnpm --filter @the-abyss/sentra-assist zip
```

The build artifacts will be in:

- Build output: `.output/chrome-mv3-prod/`
- ZIP package: `.output/sentra-assist-*.zip`

### Production Build for Firefox

```powershell
# Build for Firefox
pnpm --filter @the-abyss/sentra-assist build:firefox

# Package as ZIP for Firefox submission
pnpm --filter @the-abyss/sentra-assist zip:firefox
```

The build artifacts will be in:

- Build output: `.output/firefox-mv2-prod/`
- ZIP package: `.output/sentra-assist-*.zip`

### Verifying the Build

1. Extract the ZIP file to a temporary folder.
2. Inspect the contents to ensure no `.env` files or source maps with secrets are present.
3. Load the unpacked folder in Chrome/Firefox and perform a smoke test:
   - Open the sidepanel on an ePuskesmas page.
   - Verify login flow works.
   - Test diagnosis suggestion and forward-to-doctor features.

## Chrome Web Store

### Requirements

- A valid Chrome Web Store developer account.
- The extension's OAuth client ID configured in Google Cloud Console.
- Screenshots and promotional images (optional but recommended).
- A privacy policy URL.

### Submission Steps

1. Log in to the [Chrome Web Store Developer Dashboard](https://chrome.google.com/webstore/devconsole/).
2. Select your extension item (or click **New Item** to create one).
3. Upload the ZIP package generated by `pnpm zip`.
4. Fill in the store listing:
   - **Description:** Use the text from `wxt.config.ts` manifest description.
   - **Screenshots:** At least 1 screenshot (1280x800 or 640x400).
   - **Privacy Policy:** Link to your organization's privacy policy.
5. Click **Submit for Review**.
6. Note the review request ID for tracking.

### Post-Submission

- Typical review time: 1-7 business days.
- Monitor the developer dashboard for review status and feedback.
- Do not submit another version while one is under review unless it is critical.

## Firefox Add-ons

### Requirements

- A valid Firefox Add-on Developer account on [addons.mozilla.org](https://addons.mozilla.org/).
- The built ZIP package for Firefox.

### Submission Steps

1. Log in to the [Firefox Add-on Developer Hub](https://addons.mozilla.org/en-US/developers/).
2. Click **Submit a New Add-on** (or select an existing add-on).
3. Upload the ZIP package generated by `pnpm zip:firefox`.
4. Complete the submission form:
   - **Version notes:** Summarize changes from the previous version.
   - **Source code:** If the build process involves minification or transpilation, upload the full source code as an additional file.
5. Click **Submit Version**.

### Post-Submission

- Typical review time: 1-5 business days.
- Firefox reviews are generally more detailed; respond promptly to reviewer questions.

## Rollback Procedures

If a deployed version causes critical issues:

### Chrome Web Store

1. Log in to the Developer Dashboard.
2. Navigate to your extension.
3. Go to **Package** → select the previous stable version.
4. Click **Publish** to restore the previous version.

### Firefox Add-ons

1. Log in to the Developer Hub.
2. Navigate to your add-on.
3. Go to **Manage Status & Versions**.
4. Disable the problematic version.
5. Ensure the previous stable version remains enabled.

### Internal Rollback (Self-hosted / Enterprise)

If you distribute the extension internally:

1. Identify the last known stable ZIP package.
2. Notify users/admins to reinstall from the stable package.
3. Document the issue in `.agent/HANDOFF.md` and follow [Emergency Procedures](AGENTS.md#10-emergency-procedures).

## Environment-Specific Configuration

### Production

- `VITE_SENTRA_API_URL` must point to the production Dashboard API (HTTPS only).
- `VITE_USE_MOCK` must be `false`.
- `VITE_DEBUG` should be `false`.
- All feature flags should reflect the desired production behavior.

### Staging

- Use a staging Dashboard API endpoint.
- Enable debug logging if needed for pre-release validation.
- Keep mock data disabled unless testing isolated features.

### Development

- Point to local or development API instances.
- Enable `VITE_USE_MOCK=true` for offline development.
- Enable `VITE_DEBUG=true` for verbose logging.
